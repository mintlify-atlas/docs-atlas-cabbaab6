## POST /auth/register

Creates a new user account in the Dev-Tree system. This endpoint validates the input data, checks for existing users with the same email or handle, hashes the password securely, and stores the user in the database.

The handle is automatically converted to a URL-friendly slug format (lowercase, hyphenated) before storage.

### Request Body

<ParamField path="handle" type="string" required>
  Unique identifier for the user's profile URL. Will be converted to a slug format (e.g., "John Doe" becomes "john-doe").
  
  **Validation:** Cannot be empty
</ParamField>

<ParamField path="username" type="string" required>
  Display name for the user.
  
  **Validation:** Cannot be empty
</ParamField>

<ParamField path="email" type="string" required>
  User's email address. Must be unique across all users.
  
  **Validation:** Must be a valid email format (e.g., user@example.com)
</ParamField>

<ParamField path="password" type="string" required>
  User's password. Will be securely hashed before storage.
  
  **Validation:** Must be at least 6 characters long
</ParamField>

### Response

<ResponseField name="message" type="string">
  Success or error message describing the result of the registration attempt.
</ResponseField>

### Status Codes

- **201 Created** - User account created successfully
- **400 Bad Request** - Validation errors (missing required fields, invalid email format, or password too short)
- **409 Conflict** - Email or handle already exists in the database

### Validation Rules

The following validation rules are enforced (from `router.ts:9-14`):

1. **handle**: Must not be empty
2. **username**: Must not be empty
3. **email**: Must be a valid email format (validated using `isEmail()` from express-validator)
4. **password**: Must be at least 6 characters long

### Implementation Details

From `controllers/index.ts:10-32`, the registration process:

1. Checks if a user with the provided email already exists
2. Converts the handle to a URL-friendly slug format
3. Checks if a user with the slugified handle already exists
4. Hashes the password using bcrypt
5. Creates the user record in the database
6. Returns a success message

<Note>
  The handle is automatically converted to lowercase and spaces are replaced with hyphens. For example, "John Doe" becomes "john-doe".
</Note>

<Warning>
  Passwords are hashed using bcrypt before storage. Never store plain-text passwords.
</Warning>
